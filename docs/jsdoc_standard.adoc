This document establishes the standards for documenting code in this project using JSDoc.
The goal is to provide a clear, technical, and detailed reference for all developers.

== General Principles

. **Language**: All documentation must be written in **English**.
. **Tone**: Technical, concise, and descriptive. Avoid first-person ("I", "we").
. **Completeness**: All exported functions, classes, methods, and interfaces must be documented.

== Typing Requirement

IMPORTANT: Strict typing is **mandatory** for all methods and functions documented.

* **Inputs**: All parameters must have their types explicitly declared in JSDoc (`@param {Type} name`).
* **Outputs**: All methods/functions must explicitly declare their return type (`@returns {Type}` or `@returns {void}`).

== Classes

Classes must include a detailed description that explains their architectural purpose, responsibility, and the core problem they solve. This context is essential for understanding why the class exists and how it fits into the broader architecture, without needing repetitive phrasing like "The motivation for this class is...".

WARNING: All class public properties must be documented, explaining what they store.

=== Example

[source,typescript]
----
/**
 * Manages the game view layer and user interactions.
 *
 * Designed to decouple the rendering logic from the core game state and 
 * input processing. It orchestrates the rendering of sub-components and 
 * binds DOM events to game logic, ensuring that visual updates happen 
 * consistently in the render loop.
 */
export default class GameView {
    private p: P5;

    /**
     * The parent HTML element where the game is mounted.
     */
    public parent: HTMLElement;

    /**
     * Creates an instance of GameView.
     *
     * @param {P5} p - The P5 instance.
     * @param {HTMLElement} parent - The container element.
     */
    constructor(p: P5, parent: HTMLElement) {
        this.p = p;
        this.parent = parent;
    }
}
----

== Functions and Methods

Every function must explain **what** it does, **how** it behaves (if complex), and describing all parameters and return values.

=== Structure

. **Short Description**: One-line summary.
. **Detailed Description**: Provide a detailed base description to make its objective, context, and side-effects completely clear.
. **Cross-Referencing**: Use inline links (e.g., `{@link EntityName}`) to reference other related entities when useful.
. **@param**: One for each argument, including the type and a detailed description of its usage and constraints.
. **@throws**: Document the exceptions thrown in a detailed way, explaining the exception type and the conditions that trigger it.
. **@returns**: The return type and a detailed description of exactly what is returned.

=== Example

[source,typescript]
----
/**
 * Calculates the optimal dimensions for the game container.
 *
 * It ensures the container maintains the aspect ratio defined by constants
 * while fitting within the parent element's bounds. This is crucial for
 * supporting multiple screen sizes without distorting the game canvas.

 * @param {number} availableWidth - The maximum available width in pixels provided by the parent container. Must be greater than zero.
 * @param {number} availableHeight - The maximum available height in pixels provided by the parent container. Must be greater than zero.
 * @throws {Error} If `availableWidth` or `availableHeight` are less than or equal to zero.
 * @returns {Dimensions} An object containing the calculated optimal `width` and `height`, guaranteed to preserve the aspect ratio.
 */
function calculateDimensions(availableWidth: number, availableHeight: number): Dimensions {
    // ...
}
----

== Interfaces, Types, and Enums

Interfaces, types, and enums must include a detailed description explaining their purpose and architectural context. It should be clear why this specific data structure, type constraint, or set of values was created and how it is meant to be used. They must also document each of their properties or members.

=== Example

[source,typescript]
----
/**
 * Represents the layout response containing container references.
 *
 * Provides a standardized structure for UI components when they allocate 
 * space on the screen. By returning this structured response, the parent 
 * view can easily align other elements relatively without needing to know 
 * the implementation details.
 */
interface ButtonLayoutResponse {
    /** The container for small system buttons (e.g., Reset, Power). */
    smallButtonContainer: P5.Element;

    /** The container for directional buttons. */
    mediumButtonContainer: P5.Element;
}
----
